    Aufgabe 1.2 Grundlagen Cloud-Computing

      Wie ist der Begriff Cloud entstanden? Wieso heisst es Cloud?


      • Stammt aus Netzwerkdiagrammen, wo das Internet als Wolke dargestellt wurde.
      • Steht für virtuelle, abstrakte IT-Infrastruktur.

      Wie wird der Begriff Cloud definiert, z.B. gemäss NIST?

      Cloud Computing ist ein Modell für den allgegenwärtigen, bequemen und bedarfsgerechten Netzzugang zu einem gemeinsam genutzten Pool konfigurierbarer Computerressourcen (z. B. Netzwerke, Server, Speicher, Anwendungen und Dienste), die schnell und mit minimalem Verwaltungsaufwand oder Interaktion mit dem Dienstanbieter bereitgestellt und freigegeben werden können. Dieses Cloud-Modell besteht aus fünf wesentlichen Merkmalen, drei Dienstmodellen und vier Bereitstellungsmodellen Modellen.


      Welche Cloud Dienstleistungen kennen Sie?

      • Speicher (Dropbox, Google Drive).
      • Rechenleistung (AWS EC2, Azure VM).
      • Datenbanken (AWS RDS, Google Cloud SQL).
      • KI-Services (Google AI, AWS SageMaker).
      • SaaS (Microsoft 365, Salesforce).

      Welche Cloud Anbieter kennen Sie?

      • Grosse Anbieter: AWS, Microsoft Azure, Google Cloud.
      • Weitere: IBM Cloud, Oracle Cloud, Alibaba Cloud.

      Welche Cloud Deployment Modelle kennen Sie?

      • Public Cloud: Wird von Dritten betrieben und für viele Kunden öffentlich bereitgestellt (z. B. AWS, Google Cloud).
      • Private Cloud: Eine Cloud-Umgebung, die exklusiv für eine Organisation betrieben wird, entweder On-Premise oder bei einem Anbieter.
      • Hybrid Cloud: Kombination aus Public und Private Cloud, um Flexibilität und Sicherheit zu bieten.
      • Community Cloud: Eine spezielle Cloud für eine bestimmte Branche oder Organisationen mit ähnlichen Anforderungen.

      Was sind Cloud Service Modelle?

      • Infrastructure as a Service (IaaS): Bereitstellung von Hardware-Ressourcen (Server, Speicher, Netzwerk). Beispiel: AWS EC2, Google Compute Engine.
      • Platform as a Service (PaaS): Entwicklungs- und Laufzeitumgebungen für Anwendungen. Beispiel: Google App Engine, Microsoft Azure App Services.
      • Software as a Service (SaaS): Vollständig gehostete Anwendungen, die direkt genutzt werden können. Beispiel: Gmail, Salesforce, Microsoft 365.

      Weshalb soll ich Dienste aus der Cloud beziehen? Was sind die Vorteile?

      Vorteile der Cloud
      • Kosteneffizient (weniger Hardware).
      • Skalierbar (flexible Anpassung).
      • Überall zugänglich (Internetverbindung).
      • Automatische Updates & Sicherheit.

      Was sind die Nachteile?

      • Abhängigkeit vom Anbieter (Lock-in).
      • Datenschutz & Compliance-Herausforderungen.
      • Internetabhängigkeit (kein Netz, kein Zugriff).
      • Laufende Kosten durch Abo-Modelle.

      Welche Dienstleistungen werden in Ihrem Betrieb On-Premise (eigenes Rechenzentrum) betrieben?

      • Kritische Geschäftsanwendungen (ERP, Finanzsysteme).
      • Legacy-Systeme.
      • Interne Datenbanken mit hohen Sicherheitsanforderungen.

      Wie werden technologische Beiträge in der Cloud geteilt bzw. zur Verfügung gestellt?

      • APIs (Schnittstellen für externe Anwendungen).
      • Container & Microservices (Docker, Kubernetes).
      • Repositories & Versionierung (GitHub, GitLab).
      • CDN (Cloudflare, AWS CloudFront für schnelle Bereitstellung).
      • Serverless Computing (AWS Lambda, Azure Functions).

      Welches sind die 5 Merkmale einer Cloud?

      • On-Demand Self-Service (automatische Bereitstellung).
      • Breiter Netzwerkzugriff (überall erreichbar, es soll schnell für den user sein (Fetter Internet Zugang)).
      • Ressourcen-Pooling (geteilte Infrastruktur, Haufen Server zusammensetzen).
      • Schnelle Elastizität (flexible Skalierung sei es horizontal oder Vertikal).
      • Messbarer Dienst (verbrauchsbasierte Abrechnung, Nutzung wird gemessen und wird geholt was gebraucht werden z.B Datenbank wenn ich sie nur brauche).

    Grundlagen Container-Technologie Block 3

      Was ist Container-Technologie oder Container-Virtualisierung?

    • Container-Technologie oder Container-Virtualisierung ist eine Methode, um Software-Anwendungen und deren Abhängigkeiten so zu verpacken, dass sie überall gleich laufen – egal, ob auf einem Laptop, Server oder in der Cloud.
    • Stell dir einen Container wie eine kleine, abgeschottete Box vor, in der alles drin ist, was eine Anwendung zum Laufen braucht: der Code, Bibliotheken und Einstellungen. Dadurch kann sie unabhängig vom Betriebssystem und der Umgebung ausgeführt werden.
    • Das ist ähnlich wie bei einem Schiffscontainer: Egal, was darin transportiert wird, der Container passt überall – auf ein Schiff, einen Zug oder einen LKW. Genauso funktionieren Software-Container auf verschiedenen Systemen, ohne dass Anpassungen nötig sind.
    • Ein bekanntes Beispiel für eine Container-Technologie ist Docker.

    Was sind die Vor- und Nachteile der Container-Technologie zu virtuellen Servern (VM)?

    • Kriterium
      • Container
      • Virtuelle Maschinen (VMs)
    • Ressourcenverbrauch
      • Geringer, da Container sich den Kernel teilen
      • Höher, da jede VM ein eigenes OS benötigt
    • Startzeit
      • Sehr schnell (Sekunden)
      • Langsam (Minuten), da komplettes OS geladen wird
    • Portabilität
      • Sehr hoch, da sie überall laufen, wo eine Container-Laufzeitumgebung vorhanden ist
      • Eingeschränkt, da das gesamte OS migriert werden muss
    • Isolation
      • Gut, aber nicht so stark wie bei VMs
      • Sehr stark, da jede VM ihr eigenes OS hat
    • Sicherheit
      • Potenziell anfälliger, da alle Container denselben Kernel teilen
      • Höher, da jede VM ein isoliertes OS nutzt
    • Flexibilität
      • Hohe Skalierbarkeit und einfache Bereitstellung
      • Weniger flexibel, aber gut für traditionelle Anwendungen geeignet
    • Fazit: Container sind ideal für Microservices-Architekturen und Cloud-native Anwendungen, während VMs besser für isolierte, monolithische oder ältere Anwendungen geeignet sind.

    Welche Produkte kennen Sie im Zusammenhang mit virtuellen Servern und Container?

    • Virtuelle Maschinen (VMs):
      • VMware vSphere
      • Microsoft Hyper-V
      • KVM (Kernel-based Virtual Machine)
      • VirtualBox
      • Citrix XenServer
    • Container-Technologie:
      • Docker
      • Kubernetes
      • Podman
      • OpenShift
      • LXC (Linux Containers)
    • Unterschiede zwischen VM und Container:
      • Bereitstellung:
        • Virtuelle Maschinen (VMs): Langsame Bereitstellung (Minuten)
        • Container: Schnelle Bereitstellung (Sekunden)
      • Speicherplatz:
        • Virtuelle Maschinen (VMs): Größer, da jedes OS eigenständig läuft
        • Container: Kleiner, da Container sich das OS teilen
      • Portabilität:
        • Virtuelle Maschinen (VMs): Weniger portabel, erfordert Migration ganzer VMs
        • Container: Sehr portabel, da nur der Container benötigt wird
      • Effizienz:
        • Virtuelle Maschinen (VMs): Mehr Overhead durch OS-Instanzen
        • Container: Weniger Overhead, da nur Anwendungen virtualisiert werden
      • Betriebssystem (Kernel):
        • Virtuelle Maschinen (VMs): Jede VM hat ein eigenes OS
        • Container: Alle Container teilen denselben Kernel

      • Können virtuelle Server immer durch Container ersetzt werden?

      Container können virtuelle Maschinen (VMs) nicht immer ersetzen. Sie sind ideal für moderne, verteilte Anwendungen, die aus kleinen, unabhängigen Microservices bestehen. Container teilen sich das Betriebssystem des Hosts, was sie leicht und effizient macht. VMs hingegen bieten eine vollständige Isolation, da jede VM ihr eigenes Betriebssystem hat. Das ist wichtig für sichere oder monolithische Anwendungen, die schwer in kleine Container aufteilbar sind. Wenn eine Anwendung verschiedene Betriebssysteme benötigt, ist eine VM besser geeignet. Container starten schneller und verbrauchen weniger Ressourcen, während VMs mehr Speicher und Leistung brauchen. Für einfache Web-Apps oder Microservices sind Container perfekt. Für traditionelle, große Anwendungen oder hohe Sicherheitsanforderungen sind VMs oft die bessere Wahl. Ob Container oder VM – es kommt immer auf den Einsatzzweck an.

      Unterschied zwischen Self-Managed und Fully Managed:

      • Verwaltung:
        • Self-Managed: Unternehmen verwalten Infrastruktur selbst
        • Fully Managed: Cloud-Anbieter übernimmt Verwaltung
      • Flexibilität:
        • Self-Managed: Hohe Anpassungsmöglichkeiten
        • Fully Managed: Eingeschränkte Anpassung
      • Wartung:
        • Self-Managed: Eigenverantwortlich für Updates, Sicherheit & Skalierung
        • Fully Managed: Automatische Updates & Wartung durch Anbieter
      • Kosten:
        • Self-Managed: Meist günstiger, aber hoher administrativer Aufwand
        • Fully Managed: Teurer, aber einfacher zu betreiben
      • Beispiele:
        • Self-Managed: Eigenbetriebene Kubernetes-Cluster
        • Fully Managed: AWS Fargate, Google Kubernetes Engine (GKE), Azure Kubernetes Service (AKS)

      Fazit: Self-Managed ist gut für Unternehmen, die volle Kontrolle und Flexibilität benötigen. Fully Managed eignet sich für Firmen, die sich auf ihre Anwendungen konzentrieren wollen und weniger administrativen Aufwand wünschen.

    Aufgabe 4.1

      Warum braucht man Container-Orchestrierung?

      • Vielzahl an Containern verwalten: Wenn du viele Container (kleine, unabhängige Programme) laufen hast, wird es schwierig, jeden einzelnen manuell zu starten, zu stoppen oder zu überwachen.
      • Automatische Verwaltung: Orchestrierungstools übernehmen automatisch Aufgaben wie das Starten neuer Container, das Ersetzen von fehlerhaften Containern und das Verteilen der Last auf verschiedene Rechner.
      • Skalierbarkeit und Zuverlässigkeit: Sie helfen dabei, dass deine Anwendung auch bei hoher Last stabil und zuverlässig läuft, indem sie schnell auf Änderungen reagieren.

      Wie funktioniert Container-Orchestrierung?

      • Überwachung: Das Orchestrierungstool beobachtet ständig den Zustand aller Container.
      • Automatisches Starten und Stoppen: Es startet neue Container, wenn mehr Leistung gebraucht wird, und stoppt sie, wenn sie nicht mehr nötig sind.
      • Fehlerbehebung: Wenn ein Container ausfällt, sorgt das System dafür, dass er ersetzt oder neu gestartet wird.
      • Verteilung: Container werden intelligent auf verschiedene Rechner (Hosts) verteilt, um Ressourcen optimal zu nutzen.
      • Updates und Änderungen: Es ermöglicht auch das einfache Aktualisieren oder Ändern der Anwendung, ohne dass die ganze Anwendung stillgelegt werden muss.

      Welche Container-Orchestrierung Technologien kennen Sie?

      • Kubernetes: Das am weitesten verbreitete Open-Source-System für Container-Orchestrierung.

      Was versteht man unter "Scaling Containers"?

      • Scaling (Skalieren): Das bedeutet, die Anzahl der Container anzupassen, je nachdem, wie hoch die Nachfrage ist.
      • Horizontal Scaling: Mehr Container werden hinzugefügt, um mehr Anfragen zu verarbeiten (wie mehr Mitarbeiter einstellen, wenn viel los ist).
      • Vertical Scaling: Die Ressourcen eines einzelnen Containers werden erweitert (z. B. mehr Speicher oder CPU-Leistung), was aber oft weniger flexibel ist.
      • Einfach gesagt: Wenn deine Anwendung plötzlich sehr beliebt wird, sorgt das System dafür, dass mehr Container gestartet werden oder die Leistung sich erhöht, damit alle Nutzer schnell bedient werden können.

      Was gibt es für Deployment Strategien?

      • Rolling Update:
        • Container werden nacheinander aktualisiert.
        • Immer laufen einige alte Versionen, während neue nach und nach gestartet werden.
      • Blue-Green Deployment:
        • Es gibt zwei identische Umgebungen (z. B. „blue“ und „green“).
        • Die neue Version wird in der „green“-Umgebung gestartet.
        • Sobald alles getestet ist, wird der gesamte Traffic von „blue“ auf „green“ umgeleitet.
      • Canary Deployment:
        • Die neue Version wird zuerst nur an einen kleinen Teil der Nutzer ausgerollt.
        • Wenn alles gut funktioniert, wird sie schrittweise für alle freigeschaltet.
      • Recreate Deployment:
        • Alle alten Container werden gleichzeitig gestoppt und durch die neuen ersetzt.
        • Diese Methode kann zu kurzen Ausfallzeiten führen und wird meist nur bei weniger kritischen Anwendungen eingesetzt.
